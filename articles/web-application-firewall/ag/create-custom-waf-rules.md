# <a name="create-and-use-web-application-firewall-v2-custom-rules-on-application-gateway"></a>Creare e utilizzare regole personalizzate di Web Application Firewall v2 nel gateway applicazione

Il Web Application Firewall (WAF) v2 nel gateway applicazione di Azure offre protezione per le applicazioni Web. Questa protezione viene fornita dal set di regole di base (CRS) Open Web Application Security Project (OWASP). In alcuni casi, potrebbe essere necessario creare regole personalizzate per soddisfare esigenze specifiche. Per ulteriori informazioni sulle regole personalizzate WAF, vedere [Panoramica](custom-waf-rules-overview.md)delle regole del firewall delle applicazioni Web personalizzate .

In questo articolo vengono illustrate alcune regole personalizzate di esempio che è possibile creare e utilizzare con il WAF v2. Per informazioni su come distribuire un WAF con una regola personalizzata con Azure PowerShell, vedere Configurare le regole personalizzate di Web Application Firewall con Azure PowerShell.To learn how to deploy a WAF with a custom rule using Azure PowerShell, see [Configure Web Application Firewall custom rules using Azure PowerShell](configure-waf-custom-rules.md).

>[!NOTE]
> Se il gateway applicazione non sta usando il livello WAF, l'opzione per aggiornare il gateway applicazione al livello WAF, come visualizzato nel riquadro a destra.

## <a name="example-1"></a>Esempio 1

Sai che c'è un bot chiamato *evilbot* che si desidera bloccare la scansione del tuo sito web. In questo caso, dovrai bloccare l'User-Agent *evilbot* nelle intestazioni delle richieste.

Logica: p

```azurepowershell
$variable = New-AzApplicationGatewayFirewallMatchVariable `
   -VariableName RequestHeaders `
   -Selector User-Agent

$condition = New-AzApplicationGatewayFirewallCondition `
   -MatchVariable $variable `
   -Operator Contains `
   -MatchValue "evilbot" `
   -Transform Lowercase `
   -NegationCondition $False

$rule = New-AzApplicationGatewayFirewallCustomRule `
   -Name blockEvilBot `
   -Priority 2 `
   -RuleType MatchRule `
   -MatchCondition $condition `
   -Action Block
```

Ed ecco il JSON corrispondente:

```json
  {
    "customRules": [
      {
        "name": "blockEvilBot",
        "ruleType": "MatchRule",
        "priority": 2,
        "action": "Block",
        "matchConditions": [
          {
            "matchVariable": "RequestHeaders",
            "operator": "User-Agent",
            "matchValues": [
              "evilbot"
            ]
          }
        ]
      }
    ]
  }
```

Per visualizzare un WAF distribuito usando questa regola personalizzata, vedere [Configurare una regola personalizzata di Web Application Firewall tramite Azure PowerShell.](configure-waf-custom-rules.md)

## <a name="example-2"></a>Esempio 2

Si desidera consentire il traffico dagli Stati Uniti utilizzando l'operatore GeoMatch:

```azurepowershell
$variable = New-AzApplicationGatewayFirewallMatchVariable `
   -VariableName RemoteAddr `

$condition = New-AzApplicationGatewayFirewallCondition `
   -MatchVariable $variable `
   -Operator GeoMatch `
   -MatchValue "US" `
   -Transform Lowercase `
   -NegationCondition $False

$rule = New-AzApplicationGatewayFirewallCustomRule `
   -Name "allowUS" `
   -Priority 2 `
   -RuleType MatchRule `
   -MatchCondition $condition `
   -Action Allow
```

E il corrispondente JSON:

```json
  {
    "customRules": [
      {
        "name": "allowUS",
        "ruleType": "MatchRule",
        "priority": 2,
        "action": "Allow",
        "matchConditions": [
          {
            "matchVariable": "RemoteAddr",
            "operator": "GeoMatch",
            "matchValues": [
              "US"
            ]
          }
        ]
      }
    ]
  }
```

## <a name="example-4"></a>Esempio 4

Per questo esempio, si desidera bloccare User-Agent *evilbot*e il traffico compreso nell'intervallo 192.168.5.0/24. A tale scopo, è possibile creare due condizioni di corrispondenza separate e inserirle entrambe nella stessa regola. In questo modo si garantisce che se sia *evilbot* nell'intestazione User-Agent **che** gli indirizzi IP dell'intervallo 192.168.5.0/24 corrispondano, la richiesta viene bloccata.

Logica: p **e** q

```azurepowershell
$variable1 = New-AzApplicationGatewayFirewallMatchVariable `
   -VariableName RemoteAddr

 $variable2 = New-AzApplicationGatewayFirewallMatchVariable `
   -VariableName RequestHeaders `
   -Selector User-Agent

$condition1 = New-AzApplicationGatewayFirewallCondition `
   -MatchVariable $variable1 `
   -Operator IPMatch `
   -MatchValue "192.168.5.0/24" `
   -NegationCondition $False

$condition2 = New-AzApplicationGatewayFirewallCondition `
   -MatchVariable $variable2 `
   -Operator Contains `
   -MatchValue "evilbot" `
   -Transform Lowercase `
   -NegationCondition $False

 $rule = New-AzApplicationGatewayFirewallCustomRule `
   -Name myrule `
   -Priority 10 `
   -RuleType MatchRule `
   -MatchCondition $condition1, $condition2 `
   -Action Block
```

Ecco il codice JSON corrispondente:Here's the corresponding JSON:

              "operator": "IPMatch", 
              "negateCondition": false, 
              "matchValues": [ 
                "192.168.5.0/24" 
              ]

## <a name="example-5"></a>Esempio 5

Per questo esempio, si desidera bloccare se la richiesta non è compresa nell'intervallo di indirizzi IP *192.168.5.0/24*o se la stringa agente utente non è *in colore chrome* (ovvero l'utente non utilizza il browser Chrome). Poiché questa logica utilizza **o**, le due condizioni si trovano in regole separate, come illustrato nell'esempio seguente. *myrule1* e *myrule2* devono entrambi corrispondere per bloccare il traffico.

Logica: **not** (p **e** q) - **non** p **o non** q.

```azurepowershell
$variable1 = New-AzApplicationGatewayFirewallMatchVariable `
   -VariableName RemoteAddr

$variable2 = New-AzApplicationGatewayFirewallMatchVariable `
   -VariableName RequestHeaders `
   -Selector User-Agent

$condition1 = New-AzApplicationGatewayFirewallCondition `
   -MatchVariable $variable1 `
   -Operator IPMatch `
   -MatchValue "192.168.5.0/24" `
   -NegationCondition $True

$condition2 = New-AzApplicationGatewayFirewallCondition `
   -MatchVariable $variable2 `
   -Operator Contains `
   -MatchValue "chrome" `
   -Transform Lowercase `
   -NegationCondition $True

$rule1 = New-AzApplicationGatewayFirewallCustomRule `
   -Name myrule1 `
   -Priority 10 `
   -RuleType MatchRule `
   -MatchCondition $condition1 `
   -Action Block

$rule2 = New-AzApplicationGatewayFirewallCustomRule `
   -Name myrule2 `
   -Priority 20 `
   -RuleType MatchRule `
   -MatchCondition $condition2 `
   -Action Block
```
