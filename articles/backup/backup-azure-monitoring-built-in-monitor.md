# <a name="monitoring-azure-backup-workloads"></a>Monitoraggio dei carichi di lavoro di Backup di AzureMonitoring Azure Backup workloads

Backup di Azure offre più soluzioni di backup in base al requisito di backup e alla topologia dell'infrastruttura (locale e Azure). Qualsiasi utente di backup o amministratore dovrebbe vedere cosa sta succedendo in tutte le soluzioni e dovrebbe ricevere una notifica in scenari importanti. Questo articolo descrive in dettaglio le funzionalità di monitoraggio e notifica fornite dal servizio Backup di Azure.This article details the monitoring and notification capabilities provided by Azure Backup service.

## <a name="backup-jobs-in-recovery-services-vault"></a>Processi di backup nell'insieme di credenziali di Servizi di ripristino

Azure Backup provides in-built monitoring and alerting capabilities for workloads being protected by Azure Backup. Nelle impostazioni dell'insieme di credenziali di Servizi di ripristino, la sezione **Monitoraggio** fornisce processi e avvisi incorporati.

![Monitoraggio integrato del vault RS](media/backup-azure-monitoring-laworkspace/rs-vault-inbuiltmonitoring.png)

I processi vengono generati quando vengono eseguite operazioni quali la configurazione di backup, backup, ripristino, eliminazione di backup e così via.

I processi delle seguenti soluzioni di Backup di Azure sono illustrati di seguito:Jobs from the following Azure Backup solutions are shown here:

- Backup di macchine virtuali di Azure
- Backup dei file di AzureAzure File backup
- Backup del carico di lavoro di Azure, ad esempio SQL e SAP HANAAzure workload backup such as SQL and SAP HANA
- Agente di Backup di Azure (MAB)

I processi da System Center Data Protection Manager (SC-DPM), Microsoft Azure Backup Server (MABS) NON vengono visualizzati.

> [!NOTE]
> I carichi di lavoro di Azure, ad esempio i backup SQL e SAP HANA all'interno delle macchine virtuali di Azure, hanno un numero enorme di processi di backup. Ad esempio, i backup del log possono essere eseguiti ogni 15 minuti. Di conseguenza, per tali carichi di lavoro DB, vengono visualizzate solo le operazioni attivate dall'utente. Le operazioni di backup pianificate NON VENGONO visualizzate.

## <a name="backup-alerts-in-recovery-services-vault"></a>Avvisi di backup nell'insieme di credenziali di Servizi di ripristino

Gli avvisi sono principalmente scenari in cui gli utenti ricevono una notifica in modo che possano intraprendere azioni pertinenti. La sezione **Avvisi di backup** mostra gli avvisi generati dal servizio Backup di Azure.The Backup Alerts section shows alerts generated by Azure Backup service. Questi avvisi sono definiti dal servizio e l'utente non può creare alcun avviso personalizzato.

### <a name="alert-scenarios"></a>Scenari di avviso

Gli scenari seguenti sono definiti dal servizio come scenari avvisabili.

- Errori di backup/ripristino
- Backup completato con avvisi per l'agente di Backup di Azure (MAB)
- Ferma la protezione con mantieni la protezione dei dati/Arresta la protezione con i dati di eliminazione

### <a name="alerts-from-the-following-azure-backup-solutions-are-shown-here"></a>Gli avvisi delle seguenti soluzioni di Backup di Azure sono mostrati qui

- Backup di macchine virtuali di Azure
- Backup di File di Azure
- Backup del carico di lavoro di Azure, ad esempio SQL, SAP HANAAzure workload backups such as SQL, SAP HANA
- Agente di Backup di Azure (MAB)

> [!NOTE]
> Gli avvisi da System Center Data Protection Manager (SC-DPM), Microsoft Azure Backup Server (MABS) NON vengono visualizzati qui.

### <a name="consolidated-alerts"></a>Avvisi consolidati

Per le soluzioni di backup del carico di lavoro di Azure, ad esempio SQL e SAP HANA, i backup del log possono essere generati molto frequentemente (fino a 15 minuti in base ai criteri). Quindi è anche possibile che gli errori di backup del log siano anche molto frequenti (fino a ogni 15 minuti). In questo scenario, l'utente finale verrà sovraccaricato se viene generato un avviso per ogni occorrenza di errore. Pertanto viene inviato un avviso per la prima occorrenza e se gli errori successivi sono a causa della stessa causa principale, non vengono generati ulteriori avvisi. Il primo avviso viene aggiornato con il conteggio degli errori. Ma se l'avviso è disattivato dall'utente, l'occorrenza successiva attiverà un altro avviso e questo verrà considerato come il primo avviso per tale occorrenza. Questo è il modo in cui Backup di Azure esegue il consolidamento degli avvisi per i backup SQL e SAP HANA.

### <a name="exceptions-when-an-alert-is-not-raised"></a>Eccezioni quando non viene generato un avvisoExceptions when an alert is not raised

Esistono poche eccezioni quando non viene generato un avviso in caso di errore. ovvero:

- L'utente ha annullato in modo esplicito il processo in esecuzione
- Il processo non riesce perché è in corso un altro processo di backup (niente su cui agire qui poiché dobbiamo solo aspettare il completamento del processo precedente)
- Il processo di backup della macchina virtuale non riesce perché la macchina virtuale di Azure di cui è stato eseguito il backup non esiste più
- [Avvisi consolidati](#consolidated-alerts)

Le eccezioni di cui sopra sono progettate dalla consapevolezza che il risultato di queste operazioni (principalmente attivate dall'utente) viene visualizzato immediatamente nei client portal/PS/CLI. Così l'utente è immediatamente consapevole e non ha bisogno di una notifica.

### <a name="alert-types"></a>Tipi di avviso

In base alla gravità degli avvisi, gli avvisi possono essere definiti in tre tipi:Based on alert severity, alerts can be defined in three types:

- **Critico**: In linea di principio, qualsiasi errore di backup o ripristino (pianificato o attivato dall'utente) porterebbe alla generazione di un avviso e verrebbe visualizzato come avviso critico e anche come operazioni distruttive come eliminare il backup.
- **Avviso:** se l'operazione di backup ha esito positivo ma con pochi avvisi, vengono elencati come Avvisi di avviso.
- **Informativo**: A partire da oggi, non viene generato alcun avviso informativo dal servizio Backup di Azure.Informational : At of today, no informational alert is generated by Azure Backup service.

## <a name="notification-for-backup-alerts"></a>Notifica per gli avvisi di backup

> [!NOTE]
> La configurazione della notifica può essere eseguita solo tramite il portale di Azure.Configuration of notification can be done only through Azure Portal. Il supporto del modello PS/CLI/REST API/Azure Resource Manager non è supportato.

Una volta generato un avviso, gli utenti ricevono una notifica. Backup di Azure offre un meccanismo di notifica integrato tramite posta elettronica. È possibile specificare singoli indirizzi di posta elettronica o liste di distribuzione da notificare quando viene generato un avviso. È inoltre possibile scegliere se ricevere una notifica per ogni singolo avviso o raggrupparli in un digest orario e quindi ricevere una notifica.

![Rs Vault ha integrato la notifica e-mail](media/backup-azure-monitoring-laworkspace/rs-vault-inbuiltnotification.png)

Quando la notifica è configurata, riceverai un'e-mail di benvenuto o introduttiva. Ciò conferma che Backup di Azure può inviare messaggi di posta elettronica a questi indirizzi quando viene generato un avviso.<br>

Se la frequenza è stata impostata su un digest orario ed è stato generato un avviso e risolto entro un'ora, non farà parte del digest orario imminente.

> [!NOTE]
>
> - Se viene eseguita un'operazione distruttiva, ad esempio la protezione di arresto con i dati di **eliminazione,** viene generato un avviso e viene inviato un messaggio di posta elettronica ai proprietari, agli amministratori e ai coamministratori della sottoscrizione anche se le notifiche NON sono configurate per l'insieme di credenziali del servizio di ripristino.
> - Per configurare la notifica per i processi riusciti, utilizzare [Log Analytics](backup-azure-monitoring-use-azuremonitor.md#using-log-analytics-workspace).

## <a name="inactivating-alerts"></a>Disattivazione degli avvisi

Per disattivare/risolvere un avviso attivo, è possibile fare clic sulla voce di elenco corrispondente all'avviso che si desidera disattivare. Si apre una schermata che visualizza informazioni dettagliate sull'avviso, con un pulsante 'Inattiva' in alto. Facendo clic su questo pulsante lo stato dell'avviso viene modificato in "Inattivo". È inoltre possibile disattivare un avviso facendo clic con il pulsante destro del mouse sulla voce di elenco corrispondente a tale avviso e selezionando 'Inattiva'.

![Disattivazione degli avvisi di RS Vault](media/backup-azure-monitoring-laworkspace/vault-alert-inactivation.png)
